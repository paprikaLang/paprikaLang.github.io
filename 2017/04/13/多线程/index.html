红色皇后的"死锁"效应

通常做 TableView 轮播时,我们会在主队列的异步任务中平移tableView到它中间的位置,以便初始时就可以左右滑动
如果两个任务在同一条线程上相冲突了,也可以让其中一个任务在异步执行
例如: NSTimer 的 NSDefaultRunLoopMode 和 TableView 的 UITrackingRunLoopMode 相冲突,除了用 NSRunLoopCommonMode 兼容两种模式, 异步执行定时器也可解决;

再比如: 做直播经常会用到 Cocos2dx 来做骨骼动画, kxmovie 做视频播放, 而主线程的动画和视频是不能同时进行的,原因是两者都基于 OpenGL , 但是 OpenGL 的 context 与线程绑定,而且 Cocos2dx 必须要在主线程渲染, 所以 kxmovie 只能放到异步去操作.

可实际上 dispatch_async 在管理着一个线程池,每次调度空闲线程去执行任务, 主线程多少会被分配到, 因此最好还是用 NSThread 创建一个线程去专门做视频渲染的工作.

多个线程各司其职还好,如果它们要争夺同一个资源去读写呢?

现代操作系统在管理线程时通常采用时间片轮转算法,每个线程会被分配一段时间片,执行完这段时间片之后会被系统挂起.
原子操作可以不被系统挂起, 如果还有其他线程要操作相同的资源就只能忙等. 但是如果忙等的时间过长,等待的任务会因为超时被系统抢占时间片,甚至发生优先级反转!
郭曜源曾在博客中提到 OSSpinLock 不再安全了, 原因是低优先级线程拿到这个锁并访问共享资源之后,高优先级的线程无法进入只能忙等并占用大量 CPU ,低优先级的线程无CPU可用迟迟不能完成任务, 最终 死锁 .
除非开发者能控制所有访问锁的线程优先级相同, 否则就使用 dispatch_semaphore 信号量 来替换掉 OSSpinLock 这种自旋锁吧.

互斥锁和信号量实现的原理类似:不需要忙等, 而是阻塞线程, 睡眠, 并进行上下文切换.
有一种情况会经常发生:一个函数申请了锁之后在临界区又递归调用了自己, 但睡眠状态会使前一个锁永远无法释放.
将 pthread_mutexattr_t attr 的类型改为 PTHREAD_MUTEX_RECURSIVE 递归锁, 可以避免这种死锁发生.

@synchronized(token) 就支持递归锁, 可以递归调用 (但要注意 token 如果被其他类的 @synchronized 引用,也会死锁).

多线程之间争夺资源除了安全问题还有性能上的问题.

设想一个场景: CPU 核心正执行一个 IO 操作,然后进入等待磁盘响应状态.这个过程中CPU会不断地创建线程执行相同的任务,直到内存紧张需要释放的时候却发现大量的线程都还没有执行IO操作而无法释放.这时一旦磁盘响应了,大量线程同时开始争夺有限的 CPU 资源, 瞬间消耗的内存足够让开发者崩溃.

同样, 如果后台下载了海量数据,准备一并回传给主线程, CPU 也会被消耗殆尽, 导致程序运行逐渐变慢,操作界面陷入迟滞.

Dispatch Source 在页面刷新太频繁的时候会自动联结起来, dispatch_source_merge_data(source, 1) 的 source 按照 DISPATCH_SOURCE_TYPE_DATA_ADD 以加法计算, 并传给触发事件后执行的句柄 dispatch_source_set_event_handler.

//1、指定DISPATCH_SOURCE_TYPE_DATA_ADD，做成Dispatch Source(分派源)。设定Main Dispatch Queue 为追加处理的Dispatch Queue
dispatch_source_t source = dispatch_source_create(DISPATCH_SOURCE_TYPE_DATA_ADD, 0, 0, dispatch_get_main_queue());

__block NSUInteger totalComplete = 0;

dispatch_source_set_event_handler(source, ^{
    //当处理事件被最终执行时，计算后的数据可以通过dispatch_source_get_data来获取。这个数据的值在每次响应事件执行后会被重置，所以totalComplete的值是最终累积的值。
    NSUInteger value = dispatch_source_get_data(source);
    totalComplete += value;
    NSLog(@"进度：%@", @((CGFloat)totalComplete/100));
});

//分派源创建时默认处于暂停状态，在分派源分派处理程序之前必须先恢复。
dispatch_resume(source);

dispatch_queue_t queue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0);
for (NSUInteger index = 0; index < 100; index++) {
    dispatch_async(queue, ^{
        usleep(20000);//0.02秒
        dispatch_source_merge_data(source, 1);
    });
}
