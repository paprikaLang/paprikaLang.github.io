红色皇后的"死锁"效应

一般在做 TableView 轮播时, 都会在主队列的异步任务中平移tableView到它中间的位置,以便初始时就可以左右滑动.

不过稍作改动下,主队列的同步任务就会出现死锁的现象:

dispatch_sync(dispatch_get_main_queue(),^(void){
    NSLog(@"block is block!");
});

这是因为 block 阻塞了当前主线程又等不来下一个 runloop 重新分配线程.

如果我们在上面那段代码外面用异步封装在一个并发队列里,还会死锁吗?

dispatch_async(dispatch_get_global_queue(0, 0), ^{
    dispatch_sync(dispatch_get_main_queue(), ^{
        NSLog(@"block is over");
    });
});

不会.尽管它还是会卡在同步往主队列提交 block 的位置,不过这回由于异步不会阻塞主线程了,可以等到下一个 runloop 重新分配线程(还是主线程).
同样的思路,如果两个任务在同一条线程上相互冲突了,只要把其中一个任务放入异步线程里就可以了.例如: NSTimer 的初始化 runloop 模式和 tableView 的 UITrackingRunLoopMode 相冲突时并不是因为同一时间不可以有多个runloop模式,而是因为一条线程里只能存在一种,如果把NSTimer放到异步线程里,两者之间的冲突也就没有了.
还有一个例子是做网络直播经常会用 Cocos2dx 来做骨骼动画,用 kxmovie 做视频播放,一个常见的 bug 就是动画和视频不能同时进行,细究原因是因为两者都是基于 OpenGL 的,但是 OpenGL 的 Context 与线程是绑定的,且 Cocos2dx 必须要在主线程渲染, 解决方案是 kxmovie 在异步线程操作.

但是用 dispatch_async 把视频调到异步线程去渲染之后还是会偶尔出现视频动画不同步的情况.原因其实是 dispatch_async在管理一个线程池,每次调度空闲线程去执行任务,主线程不是不会分配的,当恰巧分配到的时候冲突就又产生了.所以还是用 NSThread 创建一个线程去专门做视频渲染的工作吧.

多个线程各司其职还好,如果它们要竞争同一个资源去读写呢? 为了保证线程安全就需要引入锁的概念了.

现代操作系统在管理线程时通常采用时间片轮转算法,每个线程会被分配一段时间片,执行完这段时间片之后
会被系统挂起,不过原子操作可以不被系统挂起,而且超过了时间片也可以一直锁到任务执行完毕.这期间如果还有其他线程要操作共享资源就只能忙等.

但是如果忙等的时间过长,等待的任务会因为超时被系统抢占时间片,更糟糕的是会出现优先级反转!
郭曜源曾在博客中提到 OSSpinLock 不再安全了,主要原因是低优先级线程拿到这个锁并访问共享资源之后,高优先级的线程无法进入只能忙等并占用大量 CPU ,导致低优先级的线程迟迟无法完成任务, 最终 死锁 .
除非开发者能控制所有访问锁的线程优先级相同否则自旋锁这类的锁最好都不要再使用了.

有一种互斥锁, 它不需要忙等, 而是阻塞线程并睡眠.

但假如一个线程在获得锁的情况下再次申请锁,会因为等待锁的释放而进入睡眠状态,这样前一个锁就永远无法释放了,还是会导致死锁.
这种情况经常发生, pthread_mutex 这个互斥锁可以封装一个类型为 PTHREAD_MUTEX_RECURSIVE 的递归锁,有了递归锁上面的死锁现象就可以避免了.
@synchronized 就是递归锁的底层再封装,所以可以嵌套使用 @synchronized ,而 dispatch_once 就不行( @synchronized(A) 中的 A 如果被其他类用来做同样的锁,也会因为相互等待对方释放锁而导致死锁的情况发生 ).

多线程之间争夺资源除了安全问题还有性能上的损耗.

设想一个场景: CPU 核心正执行一个 IO 操作,然后进入等待磁盘响应状态.这时 CPU 不会闲着,它会继续不停地创建线程执行相同的任务,直到内存足够紧张了想要释放的时候却发现大量的线程都还没有执行 IO 操作而无法释放.这时一旦磁盘响应了,大量线程同时开始争夺有限的 CPU 资源,占用的内存瞬间足够让开发者崩溃.

同样,如果此时后台从服务器下载了海量数据,要同时并发回传给主线程, CPU 也会被消耗殆尽, 导致程序运行逐渐变慢,操作界面陷入迟滞.

再来看看数据库中对应的死锁现象

数据库的事务要保证4个属性:

  • 原子性:要么100%,要么0%.如果事务崩溃状态回滚到事务之前
  • 隔离性:两个事务如果同时进行,当它俩结束的时候最终结果是一样的
  • 持久性:一旦事务提交无法改变
  • 一致性:只有合法的数据可以被存入数据库中

如果一个事务需要一条数据,先用锁把它锁住,这时再有事务需要这条数据就要等前一个事务释放这条数据,我们管这种锁叫做排它锁(有些类似自旋锁).
假设现在事务A给数据1加上排它锁并等待事务B释放数据2,而事务B在给数据2加上排它锁并等待事务1释放数据1,熟悉的相互等待,熟悉的死锁.
而它的解决办法也是简单粗暴: 超过一定时间锁如果还没加上,事务进入死锁状态,死循环链条被打破.

前不久 WCDB 第三方数据库框架开源,里面有精妙的互斥锁控制线程安全的方案:
多句柄方案
SQLITE_THREADSAFE 句柄表示写操作会先 append 到 wal 文件末尾而不是直接覆盖旧的数据,读操作会记下当前 wal 文件的状态并只访问这之前的数据
Busy Retry方案
上面的方案还没有解决多线程写写操作的阻塞,触发 Busy Handler 会让线程先休息一会儿再重复尝试

SQLite 能适配不同的平台还能支持多线程甚至多进程的原因在于它的结构:

  • Core 层:编译 SQL 语句,生成操作码控制Backend层的行为
  • Backend 层:结构中包含 B-tree, pager, OS 层(实现了存储数据的主要逻辑)

OS 层通过接口映射供对应的操作系统调用,锁的并发控制就在这一层

  • 线程锁(pthread_mutex_lock): 对应 DB 文件,通过5种状态的比较,若状态不可跳转返回 SQLITE_BUSY
  • 文件锁:对应 wal 文件,防止其它进程介入,文件锁没有线程锁类似 pthread_cond_signal 的通知机制,所以如果其它进程想要操作数据库只能不停休眠再 retry

新方案
用 pthread_mutex_lock 锁住 DB 文件,当状态不可跳转则将当前期望跳转的状态插入到一个 FIFO 队列的尾部, pthread_cond_wait 之后进入休眠状态等待 signal 唤醒
