聊聊哈希

发表于 2017-01-23

先举几个常见的有应用到哈希的例子吧:

NSDictionary和NSSet底层是一个哈希表.
@synchronized(token)中的token是通过hash算法存储在一份cache里面
苹果负责管理所有对象的引用计数和weak指针的工具–SideTables,也是一个hash表
数据库的内存缓存区中有一个缓冲池,保存着一份hash表
分类category创建的属性是保存在一个类型为AssociationHashMap的哈希映射表里的

可以看出哈希在数据结构中的影响是很大的,而本文要讨论的就是

这到底是为什么呢…….?

其实究其原因还是在于哈希算法可以快速确定元素和位置的映射关系,以O(1)的复杂度存取元素.

先举一个二叉树的例子在查询成本上做一个对比

首先二叉查找树的每个节点关键字必须满足以下的条件:
1.比保存在左子树的任何键值都要大
2.比保存在右子树的任何键值都要小

这个树有N = 15个元素,比方说我要找206
我从键值为 136 的根开始，因为
136<286，找节点136的右子树。 398="">206，找节点398的左子树
250<206，找节点250的左子树
200<206，找节点200的右子树。
但是 200 没有右子树，值不存在（因为如果存在，它会在 200 的右子树）
如果要查80,那么找到节点80 = 80之后,抽取出节点内部行的ID,再去表中查找对应的ROW ID就行了.
最后,两次查询的成本就是树内部的层数,log(N).

现在我们再看看哈希表的表现,还是举一个形象的例子:
假设我的哈希表有100个箱子,但是我现在只出10个箱子,我把要存储的元素对10取模也就是保留最后一位,再根据这个数放入10个箱子中对应的那个.取的时候再通过比较函数在一个箱子中的不同元素中找出目标元素.

比方说我要找0:

  • 哈希表计算0的哈希值是0
  • 查找第0个箱子,找到第二个元素是0;

查询过程耗费了3次运算(1次哈希运算,2次比较查找)

再比方说我要找59:

  • 哈希表计算59的哈希值为9
  • 查找9号箱，第一个找到的元素是99,不是正确的元素
  • 查找第二个元素(9)，… ，最后一个(29)
  • 元素不存在

搜索耗费了7次运算

通过上面的比较,我们可以看出哈希表的高效在于一个好的哈希算法导致元素分散的均匀程度,它能很快的定格在哪一个箱子,时间复杂度为O(1),但如果一个箱子里有很多的键值对会导致线性级访问的链表产生.

现在让我们来看一下哈希表真正的哈希运算到底是怎样的吧,以及它会如何避免这种分散不均匀的情况发生.

从哈希表的key值到映射到数组中的某一单元(也称箱子)可以分为两步:
1.prehash:从key值的任意类型转换成非负数
2.hash:将prehash所得的非负数与哈希表的箱子数量取模运算,但由于%太慢,比较常用的方法是&位运算.
这里有sideTables的算法:

最后模运算的结果范围在0–63之间,说明SideTables一共有64个”箱子”.
但如果箱子里之前已经有了其他键值对,就会发生碰撞或是冲突.冲突的解决是依靠拉链法把后来的value拼接到先来的value后面,形成一个链表.

说到冲突,就不得不提一个名词–负载因子

负载因子 = 总键值对数/箱子数

我们可以理解为负载因子越大越容易导致冲突,性能跟着下降.一般当它大于某个常数时,哈希表将会自动扩容.扩容的时机是时间和空间的权衡,既不能频繁地扩容导致性能降低,又不能一下子占用过大空间.扩容之后所有的键值对存放位置都有可能发生改变,进行重哈希.

前面提过,如果哈希函数不合理即使扩容也无法减少箱子中链表的长度而变成线性级访问.HashMap.java中给出的解决方案是当链表长度大于某一常量时转换成红黑树,提高插入和查找的性能,如果扩容长度又小于这个常量时还会变回链表;而Redis中新插入的键值对会放在箱子中链表的头部,这样省去了找链表尾部的时间,并且对于数据库来说最新插入的数据往往会被频繁使用,头插法也方便下次查找.

以上概括的介绍了哈希如何[快速确定元素和位置的映射关系].数组遍历的时间复杂度O(N)要比哈希的O(1)逊色很多,因为哈希算法的返回值NSUInteger往往和对象存储位置直接相关,而这也恰恰是数组中不适用哈希的原因—-Array当中的元素是在一片内存空间当中连续排列的,有点像链表.

[哈希算法的返回值往往和对象的存储位置直接相关],应用这一点Nginx的配置文件里可以对一个客户端的IP地址做哈希,保证同一客户端的连续Web请求会被分发到同一个服务器进行处理而保持长连接

按照这个反逻辑,我们还可以把原本有可能瞬间挤爆同一个服务器的大量用户请求通过计算某个参数的哈希值映射到不同的服务器.例如春节强红包,每个抢红包的请求发送出去的ID都不同,这样就避免了在同一个DB上读写而导致的性能下降了.
