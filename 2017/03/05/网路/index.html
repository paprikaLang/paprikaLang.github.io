一万匹"脱疆"的马

发表于 2017-03-05

朋友,读了几篇小文,还没和你打个招呼,欢迎你来到我的技术博客,我们从局域到互联,每一次与外界的联系都离不开网络.今天我们就一起讨论一下打开网页的那一刻,都发生了什么

恩…其实这一切,要从敲下那一串www.example.com的网址开始…

一. 域名解析

打开网页之前,浏览器会根据输入的网址进行DNS查询,解析域名对应的IP地址

  • 浏览器缓存:浏览器会储存DNS记录一段时间,不过储存的时间很短且容纳的缓存数量很少,所以很有可能在这里找不到缓存,如果有且没过期则解析完成
  • 系统缓存:如果上一步没有找到缓存,浏览器会搜索系统的DNS缓存
  • 路由器缓存:网络请求发出前的最后一站一般会有DNS缓存
  • ISP DNS 缓存:在网络服务运营商这儿一般都能找到相应的缓存记录,它会从根域的DNS的IP地址开始进行递归搜索，再到com域的IP地址最后是example.com域的DNS地址。把最终结果从运营商的DNS服务器到系统内核最后传递给浏览器

三. TCP握手(443端口)

  • 客户端首先发送一个连接试探ACK = 0,SYN = 1
  • 服务器收到连接请求报文之后,如果同意连接会发送确认报文SYN = 1,ACK = 1.
  • 客户端收到服务端确认后也要发送确认报文,同时可以携带要发送给服务端的数据等待加密方法的商议

四. TLS/SSL握手

建立连接之后,客户端和服务端需要商议信息传输过程中的加密协议.

  • 客户端提供协议版本号、一个客户端生成的随机数random_C、客户端支持的加密方法和压缩方法等。
  • 服务器将选择好的版本号、加密算法、压缩算法和服务器生成的随机数random_S、数字证书等返回给客户端
  • 客户端确认数字证书有效,生成一个新的随机数Premaster secret,并使用数字证书的公钥加密这个随机数.
  • 服务器使用私钥获取客户端传递过来的随机数
  • 两端商议的加密方法是使用前面的三个随机数生成对话密钥session key,用于后面通讯的对称密钥.

一些客户端想使用CDN加快访问速度,可以把私钥留在自家服务器,其他步骤交由CDN完成

为了节约RTT提高握手效率,TLS协议有缓存握手过程的机制,具体以下两种方法:

  • sessionID:重连的时候只要出示这个ID,双方就能使用先前商议好的对称密钥,不过sessionID只能保留在一台服务器上,如果重连的请求发到了另外一台服务器上就无法实现了.
  • sessionTicket:sessionTicket加密着上次握手商议好的对称密钥和加密方法,只要服务器解密后就能得到对称密钥了

TLS1.3的0-RTT会话恢复机制
SLL重连之前,客户端会重新计算服务器传回的sessionTicket,一起组成新的预共享密钥PSK(PreSharedKey).恢复会话时将PSK加密传给服务器,服务器通过会话Ticket算出PSK并解密加密数据至此完成了0-RTT会话恢复的全过程

五. http请求和响应

上传登录的账号密码需要POST请求方式,相关登录信息添加到body里面.

讲一下TCP/IP四层协议体系结构OSI对于请求数据的封装

  • http层的数据经过传输层,加上TCP的header成为一个TCP Segment;
  • 传输层的Segment经过网络层,加上IP的header,成为一个IP Packet;
  • 网路层的IP Packet经过链路层,加上Link Layer的header,成为一个frame;
  • 链路层的frame经过物理层,最后转化为物理信号

当我们已经注册过这个网站了,首次登录APP要讲用户输入的账号密码提交给服务器,服务器进行校验,若对得上则登录成功.并生产一个token,将其保存在数据库并返回给客户端.
客户端拿到token之后保存在偏好设置里,以后请求登录就把它作为参数提交给服务器校验.
服务器接到token之后会去和数据库里的token进行比较,若相同则登录成功,若不同提示用户重新登录.
每次成功登录后服务器都会重新给客户端一个新的token.
除了成功登录回调给客户端token之外还有登录状态也可以缓存,以便下次自动登录.

结尾

一万匹脱缰的马,在他脑海中旋转!
