出哈希表

OC 的 @synchronized(token) 和 Category , C++ 的 unordered_map 都有应用到 哈希表 .

class LRUCache {
private:
    unordered_map<int, LRUCacheNode*> m; //代替hash_map
    LRUCacheNode* head; 
    LRUCacheNode* tail; 
    int capacity;      
    int count;          
public:
    LRUCache(int capacity); 
    ~LRUCache();            
    int get(int key);       
    void set(int key, int value);  //未满时插入,已满时替换
private:
    void removeLRUNode();        
    void detachNode(LRUCacheNode* node);  
    void insertToFront(LRUCacheNode* node);
};



哈希表的高效取决于它的算法能否使元素分布得足够均匀.

如果分散均匀,可以很快定格一个 bucket , 时间复杂度为O(1);
但如果一个 bucket 里有很多的键值对会导致哈希碰撞和线性链表的产生(Bucket9),复杂度大大增加.

OC SideTables 的算法:

enum { StripeCount = 64 };
     ...
static unsigned int indexForPointer(const void *p) {
   uintptr_t addr = reinterpret_cast<uintptr_t>(p);
   return ((addr >> 4) ^ (addr >> 9)) % StripeCount;
}

Java HashMap 位运算:

index =  HashCode（Key） &  （Length - 1）

以 Key 等于 book 为例:
book 的十进制 Hashcode 为 3029737, 二进制就是 101110001110101110 1001
假定 HashMap 的默认长度是 16 , Length-1 的二进制为 1111
101110001110101110 1001 & 1111 = 1001
所以 index 换算成十进制等于 9.
这也解释了为什么 Java 的 HashMap 默认初始长度是 16.它的二进制各位都为 1 , index 的结果等同于 Hashcode 的后几位值(1001),效果上和 SideTables 的取模运算是一样的, 但效率要高很多.
这样,只要输入的 Hashcode 分布均匀,得到的 index 也是分布均匀的.

随着负载因子(总键值对数/箱子数)值的不断增加,哈希表的性能会逐渐下降,到达一定值时就需要扩容.
扩容的时机是时间和空间的权衡,既不能频繁地扩容导致性能降低,又不能一下子占用过大空间.
扩容之后所有的键值对存放位置都有可能发生改变,而被称为重哈希(Rehash).

由于 HashMap 是非线程安全的,在多线程下重哈希可能会引发环形链表的问题.此时如果查找一个不存在的 Key ,而这个 Key 的 Hashcode 对应的位置恰好处在环儿里,程序就会进入死循环.

Java 的 ConcurrentHashMap 是一个二级哈希结构: 上层 Segments (本身也是哈希表)之下,扩展出了若干的子哈希表 Segment .
它的设计优势在于可以不必像 HashTable 或者 Collections.synchronizedMap 一样对整个集合加锁,导致同一时间其他线程的操作被阻塞.我们可以为每个 Segment 加一把分段锁,在不同线程操作不同的 Segment 时,线程之间不受任何影响,只是同一 Segment 并发写入操作时会加上锁.

细化锁的粒度可以避免上述 HashMap 死循环的问题.

类似这种分段锁, iOS 的 SideTables 也有应用.

假如对象 A 和 B 彼此相互引用,需要一方弱引用另一方而不增加对方的引用计数来防止内存泄漏, 并且在强引用一方释放时告知它的所有弱引用(weak指针)置成 nil 以避免野指针错误.

那么 iOS 的单例又是如何释放引用的对象呢?

在 YYTextKeyboardManager 中, 需要给它的服务添加订阅者(或者叫监听者).
它的内部有一个私有属性 NSHashTable *_observers. NSHashTable 是 NSSet 的通用版本, 可以持有订阅者 observer 的弱引用.

//YYTextKeyboardManager

_observers = [[NSHashTable alloc] initWithOptions:NSPointerFunctionsWeakMemory|NSPointerFunctionsObjectPointerPersonality capacity:0];

- (void)addObserver:(id<YYTextKeyboardObserver>)observer {
    if (!observer) return;
    [_observers addObject:observer];
}

for (id<YYTextKeyboardObserver> observer in _observers.copy) {
    if ([observer respondsToSelector:@selector(keyboardChangedWithTransition:)]) {
        [observer keyboardChangedWithTransition:trans];
    }
}

这样, 订阅者只要遵守 YYTextKeyboardObserver 协议, 并在 dealloc 中取消订阅即可, 而不必在 viewDidAppear 和 viewWillDisappear 中操作, 约束调用者如何调用API并不是一种好的设计

@protocol YYTextKeyboardObserver <NSObject>
@optional
- (void)keyboardChangedWithTransition:(YYTextKeyboardTransition)transition;
@end
