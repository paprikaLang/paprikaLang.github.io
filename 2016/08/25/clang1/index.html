先提一个鸡生蛋蛋生鸡的问题:
编译器是用来编译程序的,但编译器本身也是一个程序,那它一定是由更早的编译器编译而来,所以,最早的 … ?

苹果的编译器架构分为:编译器前端和编译器后端.clang在概念上是编译器前端,同时在命令行中也作为一个”黑盒”的Driver,封装了编译管线,前端命令,LLVM命令,toolchain命令等.clang可以将代码转换成汇编代码,接着LLVM作为编译器后端将LLVM IR编译成目标平台上的本地字节码.

二. Lexical Analysis - 词法分析:

扫描器运用一种类似于有限状态机的算法把预处理过的代码文本分割成一些Token标记流,并且不校验语义

三. Semantic Analysis - 语法分析

在Clang中由Parser和Sema两个模块配合完成语法分析,验证语法层面是否正确,根据当前语言的语法生成语义节点,并将所有节点组合成抽象语法树Abstract Syntax Tree(AST)

四. Static Analysis - 静态分析

现在的语法树只做了语法层面的分析,至于是否有意义则需要静态语义分析,类型检查,找出非语法性错误,模拟代码执行路径分析出control-flow graph(CFG),在MRC时代,引用计数由程序员手动管理极易出错,点击Xcode的静态分析clang就会帮你找出对象引用计数变化的control-flow graph来检查哪里出错.

Swift 编译流通过 SILGen 生成 SIL 这一步方便做些 Swift 特定的优化，SIL 会被传递给 IR 生成阶段生成 LLVM IR，最后由 LLVM 解决余下事情。而 swift 与 C 和 OC 的交互 依靠了swift 的模块映射（Module map），它调用 Clang 的模块，将其传入 Clang importer 中生成 AST 来分析是否swift 能够和 C/OC 进行交互。

动态语义 : 在运行时才能确定的语义, 比如检查对象是否能够响应这些消息,0作为除数的报错.
静态语义 : 在编译器可以确定的语义, 通常包括声明和类型的匹配, 转换.
将clang的代码仓库克隆到本地,进入目录lib/StaticAnalyzer/Checkers,可以具体了解静态分析都在哪些方面着手,例如:ObjCUnusedIVarsChecker.cpp 是用来检查是否有定义了，但是从未使用过的变量.

五. CodeGen - IR 代码生成

CodeGen负责将语法树从上之下遍历,翻译成LLVM IR. LLVM IR 是frontend编译器前端的输出,同时也是LLVM Backend编译后端的输入,也可看做是前后端的桥接语言.
